# Review Findings — Issue #1: Per-API-key rate limiting and short link expiration

## Findings

HIGH | src/main/java/com/linkstash/controller/LinkController.java:31-32 | `incrementClick` is called before `getOriginalUrl` in the redirect handler. For expired links, `incrementClick` succeeds and persists a higher click count before `getOriginalUrl` throws `LinkExpiredException` and returns 410. The click counter is permanently corrupted for every access attempt on an expired link. | Swap the call order: call `getOriginalUrl` first (which performs the expiry check and returns the URL or throws), then call `incrementClick` only on success. Alternatively, fold the expiry check into `incrementClick` or perform both in a single `@Transactional` service method.

HIGH | src/main/java/com/linkstash/controller/LinkController.java:31-32 | No test covers the click-count side-effect of the above bug. The existing expired-link tests only assert the HTTP status/body; none verify that `clickCount` remains unchanged after a 410 response. The bug is silently undetected. | Add a test that creates an expired link, attempts a redirect (expecting 410), then fetches stats and asserts `clickCount` is still 0.

MEDIUM | src/main/java/com/linkstash/service/LinkService.java (createLink, incrementClick, deleteLink) | None of the write methods carry `@Transactional`, violating the project convention ("Use `@Transactional` on service methods that write" — CLAUDE.md). `createLink` and `incrementClick` each do a read followed by a write with no transaction boundary; under concurrent load they risk dirty reads or lost updates. This is pre-existing for `incrementClick`/`deleteLink` but the new `createLink` changes compound the risk by adding `expiresAt`. | Annotate `createLink`, `incrementClick`, and `deleteLink` with `@Transactional`.

MEDIUM | src/main/java/com/linkstash/config/WebConfig.java:16 | The interceptor is registered only on the exact path `/api/v1/links`, which means `DELETE /api/v1/links/{shortCode}` is completely unauthenticated. Any anonymous caller can delete any short link. The issue spec only requires POST to be gated, but leaving DELETE open creates an obvious denial-of-service vector. | Either accept this consciously and document it, or add API key enforcement to DELETE as well by broadening the path pattern to `/api/v1/links/**` and adjusting the method filter in `ApiKeyInterceptor`.

MEDIUM | src/main/java/com/linkstash/service/RateLimiterService.java:40-49 | `RateLimitExceededException` (an unchecked exception) is thrown from inside `ConcurrentHashMap.compute()`'s remapping function. The ConcurrentHashMap contract guarantees the exception propagates and the mapping is left unchanged, but this relies on that guarantee and makes the control flow non-obvious. A concurrent thread calling `compute` on the same key at the exact moment the exception is thrown could observe unexpected locking behaviour in some JVM implementations. | Extract the limit check before `compute()`: read the current window, decide whether to allow or reject, then call `compute()` only to perform the increment, and throw outside the lambda. This makes intent clear and avoids relying on exception-in-compute semantics.

LOW | src/main/java/com/linkstash/service/RateLimiterService.java:27 | `resetAll()` is a `public` method on a production `@Service` bean. It exists only for test isolation but is part of the public API of the service, making it callable from any Spring-managed bean including controllers. | Restrict visibility: either make it package-private, move it to a test-only subclass/extension, or annotate it with a custom `@VisibleForTesting` marker so its purpose is unambiguous.

LOW | src/main/java/com/linkstash/dto/CreateLinkRequest.java:4 | `expiresAt` accepts any `Instant` including values in the past, allowing creation of links that are expired from the moment they are stored. This produces a confusing 201 Created response for what will immediately return 410 on redirect. | Add validation: annotate `expiresAt` with `@Future` (jakarta.validation) and `@Valid` on the controller parameter, or add an explicit check in `LinkService.createLink` (`if (expiresAt != null && expiresAt.isBefore(Instant.now())) throw new IllegalArgumentException(...)`).

LOW | src/main/java/com/linkstash/interceptor/ApiKeyInterceptor.java:37 | The raw `X-API-Key` header value is passed directly to the repository query with no length cap. An attacker can send an arbitrarily long string, causing unnecessary work in the JPA layer (parameter binding, potential index scan on a very long value). Parameterized queries prevent SQL injection but not resource exhaustion. | Add a length guard before the DB call: `if (keyValue.length() > 64) { response.sendError(401, ...); return false; }` — matching the `key_value VARCHAR(64)` column limit.
