# Review Findings — Issue #1: Per-API-key Rate Limiting + Short Link Expiration

## Findings

HIGH | src/main/java/com/linkstash/controller/LinkController.java:29 | `incrementClick` is called BEFORE `getOriginalUrl` (which performs the expiry check). For an expired link the click counter is incremented and written to the DB, then the 410 is returned — expired links silently accrue phantom click counts | Move the expiry check ahead of the increment, or merge both operations into a single service method that checks expiry before writing; alternatively call `getOriginalUrl` first and only call `incrementClick` if no exception is thrown

HIGH | src/main/java/com/linkstash/ratelimit/RateLimiter.java:29 | `now` is captured outside `windows.compute(...)` then used as the new `windowStart` inside the lambda. Two threads racing at a window boundary can each see the old window as expired (T_captured >= windowStart + 60). Thread A's `compute` runs first, replaces the window. Thread B's `compute` sees `now_B >= newWindowStart + 60`? Only if it captured `now_B` >= 60 s after Thread A's new start — unlikely but the structural coupling of captured-clock-to-lambda is fragile. More concretely: if Thread A captures `now`, is paused (GC, scheduler), the window resets naturally, Thread A's compute then resets it *again* using a stale `now`, orphaning the already-accumulated counter. One request effectively escapes the limit per occurrence. | Capture the clock *inside* the `compute` lambda so `windowStart` and the expiry comparison always use the same consistent timestamp: `windows.compute(key, (k, e) -> { long t = Instant.now().getEpochSecond(); return (e == null || t >= e.windowStart + WINDOW_SECONDS) ? new WindowState(t, new AtomicInteger(0)) : e; })`; use a fresh `Instant.now()` for `retryAfter` calculation after the compute

MEDIUM | src/main/java/com/linkstash/service/LinkService.java:10,35,65,74 | `import org.springframework.transaction.annotation.Transactional` was added but `@Transactional` is not applied to any method. Write methods `createLink`, `incrementClick`, and `deleteLink` violate the project convention (CLAUDE.md: "Use `@Transactional` on service methods that write") | Annotate `createLink`, `incrementClick`, and `deleteLink` with `@Transactional`; `getOriginalUrl`, `getStats` can use `@Transactional(readOnly = true)`

MEDIUM | src/main/java/com/linkstash/service/LinkService.java:35 | No validation that `expiresAt` is in the future. A client can POST a link with `expiresAt` already in the past, creating a link that is immediately 410 on first access. This is almost certainly a user error and will be silently accepted | Add `if (request.expiresAt() != null && !request.expiresAt().isAfter(Instant.now())) throw new IllegalArgumentException("expiresAt must be in the future")` alongside the existing URL validation

MEDIUM | src/test/java/com/linkstash/controller/LinkExpiryIntegrationTest.java | No test verifies the 429 rate-limit HTTP response or the `Retry-After` response header. The unit tests for `RateLimiter` confirm the logic, but there is no end-to-end test that sends 11 POST requests with the same API key and asserts `status().isTooManyRequests()` plus `header("Retry-After")` is present and numeric | Add an integration test: loop 10 successful POST requests, make an 11th, assert 429 and a non-empty `Retry-After` header value

MEDIUM | src/main/java/com/linkstash/domain/Link.java:25,40,43 | `expiresAt` field and its getter/setter are declared with the fully-qualified type `java.time.Instant` even though `import java.time.Instant` is already present at the top of the file (used by `createdAt`). This is inconsistent with every other field in the class | Replace `java.time.Instant expiresAt` and `java.time.Instant getExpiresAt()`/`setExpiresAt(java.time.Instant ...)` with the simple name `Instant`

LOW | src/main/java/com/linkstash/ratelimit/RateLimiter.java:18 | `ConcurrentHashMap<String, WindowState>` is never pruned. Each distinct API key that has ever made a request retains a `WindowState` entry forever. For a demo with a single seeded key this is harmless, but the structure leaks memory proportional to the number of distinct keys seen since startup | Add periodic eviction of entries whose `windowStart + WINDOW_SECONDS < now`, e.g. via a scheduled task or lazily inside `tryAcquire` with a size threshold

LOW | src/main/java/com/linkstash/config/WebConfig.java:20 | `FilterRegistrationBean.addUrlPatterns("/api/v1/links")` already restricts the filter to that exact path. `ApiKeyFilter.shouldNotFilter` duplicates this guard. The double restriction is harmless today but creates a maintenance trap: if the URL pattern in `WebConfig` changes, `shouldNotFilter` may silently stop filtering correctly | Remove `shouldNotFilter` override from `ApiKeyFilter` and rely solely on `FilterRegistrationBean` URL patterns for path scoping; keep the method-check (`POST` only) inside `doFilterInternal` if DELETE on the same path should remain unauthenticated

LOW | src/main/java/com/linkstash/filter/ApiKeyFilter.java:44 | API key values are used as-is as rate-limiter map keys and are compared plaintext against the DB. In a production system raw key values should not linger in heap/thread-local memory or be stored unhashed in the DB. For this demo scope it is acceptable, but the design should be noted | Store a bcrypt/SHA-256 hash of the key in `api_keys.key_value`; hash the incoming header value before lookup and before passing to `rateLimiter.tryAcquire()`
